- Keep request validation symmetric across human and machine entrypoints: if a flag combination is invalid in the convenience surface, the raw `--request` JSON path must reject the same contradiction before planning or writing files.
- Treat `recommend from-path` as a bounded signal scan rather than a crawler: cap recursion depth, skip hidden/build/dependency trees, and prefer relevance over completeness so recommendations stay fast and useful on large repos.
- Keep the bounded scan lossy on purpose: if better recall is needed, add typed evidence from project docs, config files, and file-kind counts rather than widening recursion or reusing raw short extensions as fragment-match tokens.
- If repo intent is still under-specified after shallow path/config analysis, allow one more bounded layer from docs: read only top-level `README`-style files (plus an immediate `cli/README.md`-style surface when present), cap bytes aggressively, and map only explicit markers like `command-line`, `--json`, `stdout`, `stderr`, `TTY`, or `agent-first` into typed topic signals rather than re-tokenizing whole prose.
- Treat explicit `--json` as a hard output contract, not a best-effort hint: commands that inherently emit non-JSON artifacts should fail with a usage error instead of mixing shell code or other raw bytes into a machine-readable workflow.
- For this repo layout, treat `domains/*`, `languages/*`, `organizations/*`, `paradigms/*`, and `specialists/*` as the primary path namespace and treat leaf names like `hashimoto`, `cheney`, `lamport`, or `google` as low-confidence unless they line up with a language/tool/config signal; otherwise they will dominate `from-path` ranking with author/domain noise.
