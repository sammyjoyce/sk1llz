- If `web_search_exa` returns `401` or empty token-based failures, proceed with a plain web search tool and open only primary references tied to the current domain, then resume the workflow from the same rubric checkpoints to avoid losing review continuity.
- If no knowledge delta appears after rewriting (e.g., many introductory sentences remain), add expert-level decision heuristics before polishing prose; the common failure is satisfying format but missing practitioner-specific consequences.
- If a systems-architecture skill still reads like broad design advice after a rewrite, replace slogans with source-backed placement decisions, explicit failure thresholds, and anti-pattern consequences; these files fail when they explain architecture rather than teaching the call an experienced systems designer would make.
- If evaluation steps are blocked by missing local instructions (like absent `docs/agents`), create the directory structure first, then write only the highest-signal notes so the repository gains memory without introducing process churn.
- If `cargo run` or `cargo check` under `cli/` fails in `openssl-sys` while resolving `reqwest`, treat it as an environment or dependency-shape blocker first: either provide the system OpenSSL development package and `pkg-config` metadata, or remove the OpenSSL dependency path in favor of a more portable TLS stack before spending time on command-surface debugging.
- If `recommend from-path` looks noisy or the top result feels unrelated, check whether the scan only produced short extension tokens or fragment tag matches; the usual fix is typed signals plus stronger exact-match weights, not a wider directory crawl.
- If `recommend from-path` still surfaces author/domain skills on this repo, check whether path tokenization is admitting leaf names from namespace trees such as `domains/cli-design/hashimoto` or `languages/go/cheney`; prune those leaf tokens first and keep only reinforced signals like `rust`, `go`, `cli`, `cargo`, or `nix`.
- If `recommend from-path` under-ranks the CLI skill on a repo that is obviously CLI-shaped, inspect the top-level docs before retuning weights: missing or unrecognized markers like `command-line`, `--json`, `stdout`, `stderr`, `terminal`, or `agent-first` usually means the bounded doc pass has no strong intent signal to reinforce.
- If two skills with similar evidence swap positions unexpectedly, treat the lexicographic id tie-breaker as the first suspect and add a focused ranking fixture before changing the broader weight table.
- If PR babysit automation is missing under this repo (for example `.codex/skills/babysit-pr/scripts/gh_pr_watch.py` is absent), run the watcher from the managed home skill path (`/home/sammy/git/nix2/home/dotfiles/.agents/skills/babysit-pr/scripts/gh_pr_watch.py`) and pass `--pr auto` from the repo root so branch-to-PR inference still works.
- If `Claude Auto Review` repeatedly fails with quota errors like `You've hit your limit · resets 4am (UTC)`, classify it as an external CI blocker (not branch code), use the configured retry budget, and stop with a blocker report once retries are exhausted (`stop_exhausted_retries`).
- If a babysit push fails with `fetch first` / non-fast-forward on the PR branch, assume the remote head changed after your last fetch: inspect `git status -sb`, fetch the branch, review the incoming commits, and rebase your local fix before retrying `git push`; do not bypass the divergence with `--force`.
- If a babysat PR head changes underneath you because an external bot pushed a new commit, discard any prior "ready to merge" conclusion for the old SHA, fetch the new head, and re-run local validation plus PR-state checks against that exact commit before deciding to merge.
- If `gh pr merge` fails with `Your local changes ... would be overwritten by checkout`, the blocker is the current worktree, not the PR itself; merge from a clean detached or temp worktree, or explicitly clean the local dirt only if the user asked for that, but do not treat it as a branch-code failure.
- If merged branches are not removed by a `--merged master` cleanup pass, check `git branch -vv` for `[gone]` tracking refs; a branch can be stale after remote auto-delete even when it does not appear in the merged set yet.
- If the current branch tracks `[gone]` and still has a local-only head commit, do not assume it missed `master`: a squash-merged PR will delete the remote branch while leaving the original branch commit outside `master`, so confirm with `gh pr list/view --head <branch>` or the `origin/master` log before classifying it as stale.
