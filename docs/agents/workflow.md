# Workflow⁠‍⁠​‌​‌​​‌‌‍​‌​​‌​‌‌‍​​‌‌​​​‌‍​‌​​‌‌​​‍​​​​​​​‌‍‌​​‌‌​‌​‍‌​​​​​​​‍‌‌​​‌‌‌‌‍‌‌​​​‌​​‍‌‌‌‌‌‌​‌‍‌‌​‌​​​​‍​‌​‌‌‌‌‌‍​‌​​‌​‌‌‍​‌‌​‌​​‌‍‌‌​‌​‌‌​‍​‌​‌‌‌‌‌‍​​‌‌‌​‌​‍​​​​​​‌​‍‌​‌‌‌‌‌​‍‌‌‌‌​‌​​‍​‌‌​​‌‌​‍​​​​‌​‌​‍​​‌‌​​‌​⁠‍⁠

- For any skill rewrite task, run a strict two-pass loop: first evaluate the current file against the 8 skill-judge dimensions, then perform external research on non-obvious failure modes before writing, because rewriting against known patterns without new signal produces low D1 and low transfer value.
- In skill-judge work, keep a decision matrix for constraints vs freedom (for example, philosophy-heavy edits can stay unconstrained, while rubric-driven rewrites require explicit section coverage and required triggers), then enforce that matrix while drafting to prevent accidental over- or under-prescription.
- Treat lookup tooling failures as signals: if `web_search_exa` fails on credentials, switch to web search immediately and continue with verifiable public sources, documenting only the fallback decision rather than halting the task.
- When changing `analyze_project()`, `ingest_file_signal()`, `build_project_signals()`, or `recommend_skills()`, add focused temp-tree tests for scan depth, typed signal preservation, and rank ordering before broader validation; the common failure is silent score drift from path noise rather than a crash.
- For `recommend from-path` machine-contract changes, prefer an in-process smoke test through `cmd_recommend_from_path()` with a JSON `AppContext` and fixture manifest cache; that covers project analysis, ranking, field masking, and `score_breakdown` shape without a subprocess harness.
- When a request targets a single file under `paradigms/` or `tools/`, capture only durable process knowledge in `docs/agents/<area>/guide.md` and avoid one-off diffs in top-level logs.
- For PR babysitting, treat green CI plus `MERGEABLE/CLEAN` as necessary but not sufficient: an `isDraft=true` PR is still blocked from merge, so keep monitoring and surface draft status as the active gate rather than retrying checks or inventing code fixes.
- In PR babysitting loops, ignore watcher `process_review_comment` events that are self-authored `[codex]` status replies or self review replies; treat them as bookkeeping echoes, not actionable new review feedback.
- If a babysit fix commit fails to push with a non-fast-forward rejection, treat the remote branch as changed underneath the watcher: fetch first, inspect the remote delta, then rebase the local fix onto the current PR head before pushing again; do not force-push over unseen commits.
- If a PR head moves because a bot or autofix pushed a new commit, review and validate that exact head in a clean temp worktree before merging; do not rely on earlier green results from the previous SHA, and do not drag unrelated local dirt from the main worktree into the review loop.
- When posting PR comments containing backticks or shell-like text, avoid inline double-quoted `gh pr comment --body` strings; use single quotes, a heredoc, or `--body-file` so shell interpolation cannot mutate the message or trigger accidental local command substitutions.
- When a request is "babysit PR, else cleanup branches", resolve the current-branch PR state first (`gh pr view --json state,number,url`): if it is `MERGED`/`CLOSED` or no open PR exists, skip watcher startup and move directly to branch hygiene to avoid monitoring a terminal PR.
- For local branch hygiene after PR merge, use a conservative sequence: require a clean worktree, `git fetch --prune`, fast-forward `master` with `git pull --ff-only`, delete branches merged into `master`, then remove tracking branches marked `[gone]` in `git branch -vv`.
- If `gh pr merge --squash --delete-branch` needs to switch the local checkout and the main worktree has unrelated edits, run the merge from a clean temp worktree instead of cleaning, stashing, or risking overwrite of the live worktree.
